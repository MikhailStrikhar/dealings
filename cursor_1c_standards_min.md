# 1С: минимальные стандарты разработки для Cursor

Краткая выжимка из ССМР. Только то, что реально влияет на качество кода и архитектуры, а не на ритуальные танцы вокруг дерева метаданных.

## 1. Общие принципы

- Используй только штатные и документированные возможности платформы 1С:Предприятие.
- Не предлагай решения, которые ломают совместимость с веб-клиентом, Linux, разными СУБД и стандартными режимами работы платформы.
- Не используй модальные окна, синхронные вызовы и прочие устаревшие приемы там, где требуется асинхронное взаимодействие.
- Не отходи от стандартного поведения платформы без явной причины.
- В конфигурации не должно быть неиспользуемых объектов метаданных, процедур, функций, переменных и закомментированного мертвого кода.

## 2. Именование

- Все имена на русском, в стиле PascalCase, без транслита и сокращений без необходимости.
- Имена должны отражать смысл, а не техническую роль.
- Не используй в названиях мусорные слова вроде `Модуль`, `Функции`, `Процедуры`, `Обработчики`, если без них смысл и так ясен.
- Не используй грамматические ошибки.
- В текстах модулей не использовать букву `ё`.

### Постфиксы общих модулей

- `...Клиент` - клиентская логика.
- `...ВызовСервера` - серверные методы, доступные с клиента.
- `...КлиентСервер` - только если логика реально одинакова и на клиенте, и на сервере.
- `...Сервер` - только если нужно избежать конфликта имен или это действительно оправдано.
- `...ПолныеПрава` - привилегированный модуль.
- `...ПовтИсп` / `...КлиентПовтИсп` - повторное использование возвращаемых значений.
- `...Переопределяемый` - только для библиотечной архитектуры.

## 3. Разделение клиентского и серверного кода

- Серверную бизнес-логику держи на сервере.
- Код на клиенте минимизируй.
- Не делай лишних серверных вызовов.
- Не смешивай клиентскую и серверную логику без необходимости.
- Экспортные методы модулей `...ВызовСервера` не должны принимать и возвращать мутабельные типы (`ДокументОбъект`, `СправочникОбъект` и т.п.).
- Для обмена данными между формами и контекстами используй параметры формы и временное хранилище, а не хаотичные глобальные переменные.

## 4. Структура модулей

Используй области и фиксированный порядок разделов.

### Общий модуль

```bsl
////////////////////////////////////////////////////////////////////////////////
// Краткое описание модуля и условий его применения.
////////////////////////////////////////////////////////////////////////////////
#Область ПрограммныйИнтерфейс
#КонецОбласти

#Область СлужебныеПроцедурыИФункции
#КонецОбласти
```

### Модуль объекта / менеджера / обработки / отчета

```bsl
#Область ПрограммныйИнтерфейс
#КонецОбласти

#Область ОбработчикиСобытий
#КонецОбласти

#Область СлужебныеПроцедурыИФункции
#КонецОбласти
```

### Модуль формы

```bsl
#Область ОбработчикиСобытийФормы
#КонецОбласти

#Область ОбработчикиСобытийЭлементовШапкиФормы

#Область ОбработчикиСобытийЭлементовТаблицыФормыИмяТаблицы
#КонецОбласти

#Область ОбработчикиКомандФормы
#КонецОбласти

#Область СлужебныеПроцедурыИФункции
#КонецОбласти
```

## 5. Оформление кода

- Один оператор на строку.
- Отступы - табуляцией, размер таба стандартный.
- Длина строки - желательно до 120 символов.
- Длинные выражения переноси.
- Слева без отступа должны начинаться только объявления процедур/функций, директивы компиляции, инструкции препроцессора, объявления переменных модуля.
- Код должен быть отформатирован единообразно.

## 6. Комментарии

- Комментарии нужны только там, где без них смысл неочевиден.
- Комментарии пишутся деловым языком, по-русски.
- Не оставляй в коде отладочные комментарии, личные пометки и закомментированные куски старой логики.
- Короткий комментарий можно писать в конце строки.
- Большой комментарий пиши перед блоком кода.

Нормально:

```bsl
// Инициализируем параметры отбора для формы выбора.
```

Ненормально:

```bsl
// Вася: потом переделать
// Если что, это костыль
// старый код ниже
```

## 7. Процедуры и функции

- Процедура делает действие, функция возвращает значение.
- Имена процедур и функций должны отражать результат или действие.
- Экспортный интерфейс располагай вверху модуля.
- Служебные методы не делай экспортными без причины.
- Не делай одну процедуру обработчиком нескольких событий формы. Для каждого события - свой обработчик, а общую логику выноси в отдельную служебную процедуру.

## 8. Обработка ошибок

- Не глотай исключения молча.
- Если перехватил исключение, либо обработай его осмысленно, либо запиши в журнал регистрации и пробрось дальше.
- Не используй `Попытка ... Исключение` для проверки существования реквизита, макета, метода и т.п. Для этого используй метаданные и явные проверки.
- Не используй исключения для преобразования типов. Используй `ОписаниеТипов`.

Правильно:

```bsl
НачатьТранзакцию();
Попытка
    // работа с данными
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации(
        НСтр("ru = 'Выполнение операции'"),
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ПодробноеПредставлениеОшибки(ИнформацияОбОшибке())
    );
    ВызватьИсключение;
КонецПопытки;
```

## 9. Транзакции и блокировки

- Все вызовы `НачатьТранзакцию()` должны иметь парный `ЗафиксироватьТранзакцию()` или `ОтменитьТранзакцию()`.
- Внутри транзакции не должно быть лишнего кода, UI и побочных действий.
- Блокировки должны быть минимально достаточными по области и времени.
- Не затягивай транзакции длинными вычислениями, обращением к UI, сети и файловой системе.

## 10. Формы

- На форме держи только логику интерфейса.
- Бизнес-логику выноси из формы.
- Параметры формы инициализируй явно.
- Для каждого события элемента формы создавай отдельный обработчик.
- Повторяющийся код обработчиков выноси в служебные процедуры.

## 11. Работа с данными и метаданными

- Выбирай объект метаданных по смыслу: документы для событий, регистры для хранения фактов и состояний, справочники для сущностей.
- Не создавай составные типы и универсальные сущности "на всякий случай", если у этого нет прикладного смысла.
- Не используй слишком общие имена таблиц, алиасов и сущностей в запросах вроде `Документ`, `Справочник`, `Сведения` без необходимости.

## 12. Запросы

- Запрос должен быть читаемым.
- Не тащи лишние поля, таблицы и соединения.
- Фильтрацию делай как можно раньше.
- Избегай сложных конструкций, если можно выразить логику проще.
- Временные таблицы и пакетные запросы используй осознанно, когда они реально упрощают или ускоряют решение.
- Пиши запрос так, чтобы его можно было сопровождать без спиритического сеанса.

## 13. Проверка заполнения и валидация

- Где реквизит реально обязателен для пользователя, используй свойство `Проверка заполнения`.
- Не прячь обязательность полей глубоко в коде без необходимости.
- Сложные проверки, зависящие от условий, оформляй в коде явно и понятно.

## 14. Что должен делать Cursor при генерации кода

Когда генерируешь код 1С, придерживайся таких правил:

1. Сначала выбери правильный контекст выполнения: клиент, сервер, вызов сервера, клиент-сервер.
2. Соблюдай структуру модуля через `#Область`.
3. Разделяй публичный интерфейс и внутреннюю реализацию.
4. Не добавляй лишние экспортные методы.
5. Не оставляй мертвый код, временные комментарии и отладочные конструкции.
6. Не используй исключения вместо нормальной логики ветвления.
7. Для транзакций всегда показывай корректную схему `Попытка/Исключение`.
8. Для формы не смешивай UI и бизнес-логику.
9. Для серверных вызовов не передавай мутабельные объекты между клиентом и сервером.
10. Пиши код простым, читаемым и поддерживаемым.

## 15. Мини-чеклист перед коммитом

- Имена понятные и единообразные.
- Нет мертвого кода и закомментированных кусков.
- Нет лишнего `Экспорт`.
- Нет лишней логики на клиенте.
- Обработчики формы не раздуты.
- Исключения не проглатываются.
- Транзакции парные.
- Код разбит на области.
- Комментарии только по делу.
- Код можно читать без телепатии.
